# Managing Mom's Recipes in GitHub

This document covers what to commit, what to ignore, branching strategy, and the day-to-day workflow for keeping the repo in sync with your local development and Namecheap deployments.

---

```

├── .gitignore                      ← see below

├── recipe_pipeline.py              ← COMMIT — main deploy tool

├── input/                          ← OPTIONAL — Word docs (can be large)
```

## .gitignore

Create this at the repo root:

```gitignore
# Credentials — never commit
pipeline_config.json

# Generated output — rebuilt by --process and --build
output/

# Database backups
backups/

# Python
__pycache__/
*.pyc
*.pyo
.venv/
venv/

# macOS
.DS_Store

# Windows
Thumbs.db

# SQLite databases
*.db
*.sqlite
*.sqlite3

# Logs
*.log

# Editor
.vscode/
.idea/
*.swp
```

### Optional: exclude input/ Word docs

If your Word docs are large (they likely are with embedded images), add:

```gitignore
# Word documents — stored separately, not in git
input/
```

If you want input tracked (useful for sharing with family), leave it out of `.gitignore`. Be aware `.docx` files with images can be 5–20 MB each, and you have 300+ recipes — that's potentially hundreds of MB of binary files. Consider [Git LFS](https://git-lfs.com) if you go that route.

## Initial Setup

```bash
# Clone the repo on a new machine
git clone https://github.com/yourname/momsrecipes.git
cd momsrecipes

# Install Python dependencies
pip3 install python-docx Pillow lxml requests

# Set up credentials (creates pipeline_config.json — not committed)
python3 recipe_pipeline.py --setup

# Verify connection
python3 recipe_pipeline.py --check
```

## Branching Strategy

### Simple (recommended for a personal/family project)

```
main        ← production-ready code, always deployable
dev         ← active development
```

Work on `dev`, test on staging, merge to `main`, then promote to production.

```bash
# Start new work
git checkout dev
git pull origin dev

# ... make changes ...

git add -A
git commit -m "Fix drag-and-drop reorder on recipe pages"
git push origin dev

# When ready to release
git checkout main
git merge dev
git push origin main

# Deploy to staging and promote to production
python3 recipe_pipeline.py --deploy-api --update-db
python3 recipe_pipeline.py --promote
```

### Feature branches (if multiple people contribute)

```
main        ← production
dev         ← integration
feature/*   ← individual features
```

## Day-to-Day Workflows

### Editing frontend code (CSS, JS, HTML)

```bash
git checkout dev

# Edit files in src/frontend/
# ...

python3 recipe_pipeline.py --build            # inject API URL + cache-bust
python3 recipe_pipeline.py --deploy-frontend  # push to staging

# Test on staging, then commit
git add src/
git commit -m "Update recipe card styles"
git push origin dev
```

### Editing API (PHP files)

```bash
git checkout dev

# Edit files in src/api/
# ...

python3 recipe_pipeline.py --deploy-api  # push to staging

# Test, then commit
git add src/api/
git commit -m "Fix get_recipe UUID resolution"
git push origin dev
```

### Processing new/updated Word docs

```bash
# Add/update .docx files in input/
python3 recipe_pipeline.py --process    # extract recipes locally
python3 recipe_pipeline.py --update-db  # push to staging DB + upload HTML/images

# If input/ is tracked in git
git add input/
git commit -m "Add Norwegian recipes from Aunt Ingrid"
git push origin dev
```

### Promoting staging to production

```bash
# After testing on staging
git checkout main
git merge dev
git push origin main

python3 recipe_pipeline.py --promote
```

### Updating approved users

```bash
# Edit approved_users.json
python3 recipe_pipeline.py --update-users

git add approved_users.json
git commit -m "Add family member login"
git push origin dev
```


## Sensitive Files Checklist

Before every `git push`, confirm these are never staged:

```bash
git status
```

These should **never** appear as staged or committed:
- `pipeline_config.json` — FTP credentials
- `backups/` — database snapshots
- `output/` — generated files
- `*.db` — SQLite databases

If you accidentally committed credentials:

```bash
# Remove from tracking (keeps the file locally)
git rm --cached pipeline_config.json
git commit -m "Remove accidentally committed credentials"
git push

# Then rotate your FTP password immediately on Namecheap
```

## Keeping a Second Machine in Sync

```bash
git pull origin dev

# pipeline_config.json is not in git — recreate it
python3 recipe_pipeline.py --setup

# output/ is not in git — rebuild from source
python3 recipe_pipeline.py --process
python3 recipe_pipeline.py --build
```

## GitHub Actions (Optional)

If you want automatic deployment when you push to `main`, create `.github/workflows/deploy.yml`:

```yaml
name: Deploy to Staging

on:
  push:
    branches: [main]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3

      - name: Set up Python
        uses: actions/setup-python@v4
        with:
          python-version: '3.11'

      - name: Install dependencies
        run: pip install python-docx Pillow lxml requests

      - name: Create pipeline config
        run: |
          echo '{
            "ftp_user": "${{ secrets.FTP_USER }}",
            "ftp_pass": "${{ secrets.FTP_PASS }}",
            "ftp_remote_dir": "${{ secrets.FTP_REMOTE_DIR }}",
            "prod_dir": "${{ secrets.PROD_DIR }}",
            "api_url": "${{ secrets.API_URL }}"
          }' > pipeline_config.json

      - name: Deploy API
        run: python3 recipe_pipeline.py --deploy-api

      - name: Deploy frontend
        run: python3 recipe_pipeline.py --build --deploy-frontend
```

Store `FTP_USER`, `FTP_PASS`, `FTP_REMOTE_DIR`, `PROD_DIR`, and `API_URL` as [GitHub repository secrets](https://docs.github.com/en/actions/security-guides/encrypted-secrets).

> Note: `--process` and `--update-db` are not included in the action above — Word doc processing and DB updates are intentionally manual steps since they require the `input/` files and should be reviewed before pushing.
